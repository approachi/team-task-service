# Сопроводительное письмо

## Выбор технологий

### Go 1.25
Последняя стабильная версия на момент разработки. Для нового проекта нет причин стартовать на старой версии — берём последнюю и получаем улучшения рантайма, GC и стандартной библиотеки бесплатно.

### Роутер: `chi`
Chi работает поверх стандартного `net/http` — хендлеры и middleware остаются обычным Go-кодом без привязки к фреймворку, что позволяет юнит-тестировать middleware изолированно, без специфичного для фреймворка контекста запроса. При росте API и необходимости встроенной валидации — `gin` или `echo`, при экстремальных требованиях к производительности — `fiber`, но ценой несовместимости с `net/http`.

### Доступ к БД: `database/sql` + `jmoiron/sqlx`, без ORM
ТЗ требует написанные вручную JOIN'ы, оконную функцию и запрос на проверку ссылочной целостности — это требует полного контроля над генерируемым SQL. `sqlc` добавил бы шаг кодогенерации, лишний, пока схема ещё меняется; сложные запросы в `GORM` всё равно пришлось бы писать через `Raw()`, так что ORM не даёт выгоды, зато добавляет свои соглашения (хуки, магический вывод колонок).

### Миграции: `pressly/goose`
Функционально эквивалентен `golang-migrate`; выбран из-за более простого CLI и опциональной поддержки миграций на Go-коде, если понадобится миграция, невыразимая обычным SQL.

### Тестирование: `testify/require` + собственные fakes, `testcontainers-go` для интеграционных
`require`, а не `assert` — упавшее предусловие должно немедленно останавливать тест. Fakes для `UserRepository`/`TeamRepository`/`TaskRepository`/`Notifier` написаны вручную — интерфейсов немного и они стабильны, кодогенерация мока ради них была бы лишним слоем. `testcontainers-go` — для интеграционного теста с реальным MySQL, там, где важны настоящие ограничения БД (уникальные ключи, поведение FK, атомарность транзакции обновление+история под блокировками строк).

### Ключевое техническое решение: инвалидация кеша списка задач

Список задач команды кешируется в Redis (cache-aside, TTL 5 мин). Инвалидация — не удаление ключей по паттерну, а версионный счётчик на команду:

```
INCR tasks:list:version:{team_id}
```

Каждый ключ кеша встраивает текущую версию команды. При изменении задачи счётчик увеличивается на единицу — O(1) вместо `SCAN`/`KEYS` + `DEL` по всем закешированным комбинациям фильтров и страниц (O(n) и небезопасно под нагрузкой). Устаревшие записи с прошлой версией просто становятся недостижимыми и самостоятельно истекают по TTL, без явного удаления.

## Что было бы добавлено в production

### Инфраструктура кода

- **CI/CD**: в репозитории нет CI-пайплайна — весь прогон тестов/линта сейчас ручной, через `make test`/`make lint`. Нужен PR-пайплайн (`go vet` → `golangci-lint` → `go test -race` с `testcontainers-go`, значит раннеру нужен Docker → `govulncheck`), сборка образа с тегом = SHA коммита, и миграции как отдельный явный шаг деплоя. Перед реальным `goose up` — dry-run на копии прод-схемы (снапшот staging-БД, накатить и сразу откатить в CI): это ловит синтаксические ошибки и долгие блокировки на прод-объёме данных до раската, а не просто печатает SQL, как `goose up --dry-run` без реальной БД. Сам `goose up` — только после этого и перед раскаткой, с обратно совместимыми (purely additive) миграциями на время rolling-деплоя.
- **Contract-тест Swagger vs факт**: `swag init` генерирует спеку из аннотаций рядом с хендлерами, но ничто не проверяет автоматически, что аннотация не разошлась с реальным телом ответа.
- **Параллельные тесты**: добавить `t.Parallel()` в юнит-тесты; для интеграционных — сначала перейти на транзакционную изоляцию (rollback вместо пересоздания схемы), затем `t.Parallel()`.

### Надёжность и безопасность

Уже сделано: `ReadTimeout`/`WriteTimeout`/`IdleTimeout` на HTTP-сервере, `http.MaxBytesReader` на тело запроса, rate limiting на Redis (по пользователю и отдельно по IP для `/register`/`/login`, fail-open при недоступном Redis), circuit breaker (`sony/gobreaker`) над отправкой приглашений.

Чего не хватает:
- **Отзыв токенов / logout** — JWT сейчас нельзя инвалидировать раньше TTL (24ч). Нужен короткий access-токен + refresh с ротацией и allowlist активных `jti` в Redis, чтобы logout и принудительный отзыв работали немедленно.
- **Блокировка аккаунта** после N неудачных попыток входа — IP-лимитер уже закрывает грубый брутфорс с одного адреса, но не медленный распределённый перебор на один и тот же аккаунт.
- **CORS и security-заголовки** (`X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`) — сейчас их нет вовсе.
- **Секреты** — сейчас обычные env-переменные; в проде секрет-менеджер (Vault/AWS Secrets Manager) вместо голых env, с ротацией `JWT_SECRET`.
- **Сканирование уязвимостей**: `govulncheck` для зависимостей, `trivy` для Docker-образа — в CI на каждый PR.
- **Аудит действий с правами** — `task_history` фиксирует изменения полей задачи, но не изменения роли участника или исключение из команды.

### Observability: золотой треугольник

Уже сделано: структурные JSON-логи через `slog`, три серии метрик Prometheus с предсказуемой кардинальностью (`RoutePattern()`, а не сырой путь), `request_id` в контексте на каждый запрос.

Чего не хватает:
- **Логи**: агрегация через Loki или managed-вариант облака — JSON в stdout некому собирать при нескольких инстансах.
- **Трейсинг**: OpenTelemetry — span от HTTP-хендлера через Redis-кеш до SQL-запроса, с тем же `request_id`, чтобы логи и трейсы были связаны.
- **Дашборды и алерты**: error rate и p95/p99 латентность по метрикам уже есть в сырых данных, но нет ни Grafana-дашборда, ни Alertmanager; отдельно стоило бы экспортировать состояние circuit breaker как gauge.
- **SLO** — без зафиксированного целевого уровня метрики есть, а решения "это инцидент или нет" — нет.

### Функциональность

- Подтверждение email при регистрации.
- Восстановление пароля по одноразовой ссылке.
- Мягкое удаление задач и исключение участника из команды — сейчас в API нет ни `DELETE /tasks/{id}`, ни выхода/исключения из команды.
- Асинхронная отправка приглашения через очередь вместо синхронной отправки в теле запроса — сейчас circuit breaker защищает от каскадного отказа, но не от того, что реальный email-провайдер отвечает медленнее, чем нужно пользователю ждать ответ API.

## Инфраструктура для production

**Kubernetes:**
- 2–5 реплик за Load Balancer, автоскейлинг через `HorizontalPodAutoscaler` (можно завести на `http_requests_total` вместо голого CPU)
- Сервис уже спроектирован stateless-дружелюбно — роль не кешируется в JWT, список задач кешируется в Redis по версионному счётчику, а не в памяти процесса — горизонтальное масштабирование не требует переделок
- `readinessProbe`/`livenessProbe` поверх уже существующего graceful shutdown
- Rolling-деплой требует обратно совместимых миграций (см. выше)

**База данных:**
- Managed MySQL (RDS Multi-AZ) вместо самостоятельно управляемой — automated backups и point-in-time recovery из коробки
- Стоит рассмотреть и PostgreSQL вместо MySQL: строже стандартный SQL, `EXPLAIN (ANALYZE, BUFFERS)` удобнее для разбора тяжёлых аналитических запросов ниже, лучше ведёт себя под смешанной OLTP+аналитической нагрузкой на одной БД. Цена перехода некритичная, но не нулевая: `internal/repository/mysqlerr.go` завязан на код ошибки MySQL-драйвера (`1062` — duplicate entry), а `NOW() - INTERVAL 7 DAY` в `analytics_repository.go` — MySQL-синтаксис интервала (в Postgres `INTERVAL '7 days'`)
- ElastiCache для Redis — кеш и rate limiter уже спроектированы как fail-open при недоступности Redis, так что требование к его доступности мягче, чем к MySQL, но failover и патчинг всё равно не то, что стоит писать руками
- Read-реплика для трёх аналитических отчётов (JOIN трёх таблиц, оконная функция, `NOT EXISTS`) — самых тяжёлых запросов в сервисе — по мере роста нагрузки

**Очередь сообщений:**
- SQS (или RabbitMQ вне облака) для асинхронной отправки приглашений — нагрузка "одно событие на один invite" — это command queue, а не event stream с множеством consumer'ов и требованием replay, так что Kafka здесь была бы инфраструктурным оверинжинирингом

**CI/CD:**
- GitLab CI — пайплайн описан выше; интеграционным тестам на `testcontainers-go` нужен Docker-in-Docker (`services: [docker:dind]` с `privileged`-executor), а shared-раннеры GitLab.com privileged-режим не дают — нужен self-hosted раннер (Kubernetes executor с DinD либо Docker executor с примонтированным `/var/run/docker.sock`)
- Отдельный protected-раннер под деплой-стадии (`goose up`, `kubectl apply`) с доступом к прод-секретам — обычные раннеры этот доступ иметь не должны

**Наблюдаемость:**
- Prometheus + Grafana + Loki + Tempo — продолжение уже начатого стека (метрики уже на `client_golang`), а не переход на SaaS-конкурента, который означал бы переписывать рабочую инструментацию без функциональной необходимости

**Секреты:**
- Облачный secret manager (AWS Secrets Manager/Parameter Store) или Vault при мультиоблачности — не требует изменений в `internal/config`, секрет-менеджер просто становится источником тех же env-переменных на старте пода

**Infrastructure as Code:**
- Terraform для всего перечисленного выше
